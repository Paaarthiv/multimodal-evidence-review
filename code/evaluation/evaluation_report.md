## 4. Operational analysis

- **Architecture:** exactly **one vision call per claim** (all images of a case batched into a single call). ~0.00 calls/row.
- **Model calls:** sample run = 0 calls for 20 rows (20 cache hits). Projected test run ≈ **0 calls** for 44 rows.
- **Tokens:** sample = 0 in / 0 out (~0/0 per call). Projected test ≈ 0 in / 0 out.
- **Images processed:** 31 (sample) + 85 (test) = 116 total; downscaled to ≤512px longest edge before upload to cut image tokens.
- **Approximate cost (assumed model `gemini-2.5-flash`, $0.3/M in, $2.5/M out):** sample ≈ $0.0, **full test set ≈ $0.0**.
- **Latency / runtime:** sample took 0.4s (~0.0s/row at concurrency); full test ≈ 1s wall-clock with parallelism.
- **Rate limits / reliability strategy:**
  - Final run is **fully local (Ollama)** → no API rate limits and **$0 cost**; `MAX_CONCURRENCY=1` because a single local model cannot truly parallelize.
  - Provider/model failover is built in (`PROVIDER_ORDER`, e.g. local → Gemini → OpenAI) for the cloud path; quota/daily-cap errors fail over immediately, per-minute/5xx errors retry with exponential backoff.
  - On-disk response cache keyed by (provider, model, prompt, image bytes) so re-runs, eval iterations, and resumed/interrupted runs never repay for identical calls (also enables crash-safe resume of the long local run).
  - Local-model tuning for tractable CPU inference: plain JSON mode (not schema-grammar), compact output contract, `num_ctx` sized for multi-image, and 512px image downscaling.

> Measured figures above come from the actual run; if this report was regenerated from cache, see `code/logs/last_run.json` for the real test-set call/token/latency numbers.
